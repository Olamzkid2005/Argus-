## Guiding Principle

The dependency graph in section 9 is the primary constraint. You cannot build the planner before the workflow registry, cannot build the evidence store before the schema, cannot wire commands before the modules they orchestrate. The backend remediation runs in parallel with frontend foundation work since the two codebases are independent until the MCP bridge is tested end-to-end.

## Phase 0 — Foundations (do all of these before writing any module code)

These are pure definition tasks with no dependencies between them, so they can be done in any order or in parallel. None of them produce runnable code, but every subsequent phase depends on them being complete.

**0.1 — Wire protocol spec**
Write the MCP JSON-RPC wire protocol document before touching either `mcp_transport.py` or `mcp-client.ts`. This is the handshake that the Python and TypeScript sides must agree on. Every other task that touches the bridge depends on this being settled.

**0.2 — Type definitions**
Define all shared types in their respective files. This means `Capability` enum in `capabilities.ts`, `Confidence` enum and `Finding` in `engagement/types.ts`, `NormalizedFinding` and `Severity` in `reporting/types.ts`, `EngagementState` and `PhaseStatus` in `engagement/types.ts`, and `VerificationScenario` and `VerifierResult` in `browser/types.ts`. These are Tasks 0.0.1, 0.8, and 0.13 combined. Do them together since the types cross-reference each other.

**0.3 — Credential interface**
Define the `CredentialStore` interface, the JSON credential file schema, and the zeroization contract. This is Task 0.0.4 and 0.3 combined.

**0.4 — Error recovery matrix**
Define `ErrorRecovery` type and `PHASE_ERROR_POLICY` per phase. Task 0.4.

**0.5 — PlannerContext interface**
Define `PlannerContext` with `findings`, `executedCapabilities`, `insertedPhases`, `replanCount`, and the serializable vs in-memory boundary explicitly documented. Task 0.12.

**0.6 — Approval gates framework**
Define the policy YAML format and `ApprovalGate` interface. Task 0.9.

**0.7 — ADR documents**
Write all 16 ADRs. Task 0.7. These should be written now while the decisions are fresh, not retrofitted later.

## Phase 2 — Core Modules (strict order matters here)

The document's recommended order (2.5 → 2.6 → 2.1 → 2.2 → 2.3 → 2.4) is correct but needs one insertion.

**Step 1 — Workflow Registry (Task 2.5)**
Build `src/argus/workflows/` first: registry, loader, YAML validation against the `Capability` enum, all five built-in workflow YAMLs, `ToolRegistry` with scoring-based ranking. Nothing else can be built correctly without the registry, because the planner selects tools by querying it.

**Step 2 — Engagement State Store (Task 2.6)**
Build `src/argus/engagement/`: Drizzle schema (already defined in Phase 1), `EngagementStore`, `ConfidenceEngine` with the corrected promotion logic (the missing case from gap 2), `recovers.ts` with workflow version validation. The store is the persistence layer everything else writes to.

**Step 3 — MCP Bridge (Task 1.5 + Task 0.11)**
Build `src/argus/bridge/`: `mcp-client.ts` implementing the wire protocol, `WorkerSupervisor` with reset-on-stability logic, child process lifecycle management (SIGTERM/SIGINT forwarding, graceful shutdown). Also implement drift detection (Task 0.0.5). Do this before the planner because the planner calls tools through the bridge.

**Step 4 — Planner (Task 2.1)**
Build `src/argus/planner/`: all types, `PlannerContext`, `WorkflowPlanner.plan()` using `WorkflowRegistry` and `ToolRegistry`, `replan()` with all three termination safeguards, `planDeterministic()` fallback, `replan-rules.ts` with documented insertable vs initial-only capabilities, `InProcessExecutor` with per-tool timeout enforcement.

**Step 5 — Evidence Engine (Task 2.2)**
Build `src/argus/evidence/`: `EvidenceCollector` with storage limits enforcing active-engagement exclusion from pruning, `ArtifactStore` backed by Drizzle, `EvidenceManifest` with `package_hash`, `/evidence prune`, `/verify-package`. Add the `audit_log` table to the schema and wire it up here.

**Step 6 — Reporting (Task 2.3)**
Build `src/argus/reporting/`: `normalizer.ts` converting tool-specific output to `NormalizedFinding`, report generator for markdown/SARIF/JSON. This comes after evidence because reports query the evidence store.

**Step 7 — Browser Module (Task 2.4)**
Build `src/argus/browser/` last because it depends on everything else: evidence collection for artifacts, engagement store for findings, credential store for roles. Implement `VerificationScenario` interface, `PlaywrightEngine`, `ObserveLoop`, and the three verifiers (BOLA, StoredXSS, PrivilegeEscalation).

## Critical Path Summary

The tasks that gate everything else, in order:

1. Wire protocol spec → unlocks both `mcp_transport.py` and `mcp-client.ts`
2. Type definitions → unlocks every module that references shared types
3. SQLite schema with Drizzle → unlocks every module that persists state
4. Backend B.0 remediation → unlocks meaningful MCP bridge testing
5. Workflow Registry → unlocks planner
6. Engagement Store → unlocks everything that needs to persist findings
7. MCP Bridge → unlocks planner tool execution
8. Planner → unlocks `/assess`
9. `/doctor` → validates the assembled system before real scans

Everything else flows from those nine gates in order.
