# Tool Registry Investigation: Findings

**Date:** 2026-07-20  
**Author:** AI audit cross-check  
**Status:** Resolved — not a duplicate-registry problem

## Background

The Strengthening Plan review flagged a concern: `tool_definitions.py` and
`tools/definitions/*.yaml` + `_generated_tools.py` both appeared to claim to be
"the single source of truth" for tool metadata. The concern was that these might
be duplicate, drifting registries.

## Architecture: Layered, Not Duplicated

After file-by-file investigation, the Python-side tool-registry is an
**intentionally layered architecture** with four layers:

```
Layer 1:  YAML source of truth
          tools/definitions/*.yaml  (65 files)
          │
          ▼ (scripts/generate_tool_defs.py)
Layer 2:  Auto-generated Python
          _generated_tools.py  (1155 lines, 60 tools)
          │
Layer 3:  from _generated_tools import *  (line 210)
          + 60 inline _register() calls that can override generated defs
          tool_definitions.py  (1444 lines)
          │
          ▼ imports TOOLS dict
Layer 4:  PATH-scanning consumer
          tool_core/registry.py  (172 lines)
```

### Layer 1: YAML Definitions (`tools/definitions/*.yaml`)

65 YAML files, each defining one tool's execution metadata:
- `name`, `description`, `phases`, `parameters`
- `timeout`, `signal_quality`, `priority`, `cost`, `risk_level`
- `capabilities`, `enabled` flag

Example (`register.yaml`):
```yaml
name: register
command: python3
args: ["-m", "argus-workers.tools.run_agent_tool", "register"]
description: "register - security assessment tool"
phases: ["recon"]
timeout: 300
priority: 95
```

### Layer 2: Generated Python (`_generated_tools.py`)

Auto-generated by `scripts/generate_tool_defs.py`. Contains `_register()` calls
for all 60 tools. Header states:
```python
"""
Auto-generated tool registrations — DO NOT EDIT BY HAND.
Generated by: scripts/generate_tool_defs.py
Source:      tools/definitions/*.yaml
"""
```

### Layer 3: Public API (`tool_definitions.py`)

Imports generated definitions at line 210:
```python
from _generated_tools import *  # noqa: F403,E402
```

Then adds **60 inline `_register()` calls** that can intentionally override
YAML-generated definitions with richer metadata (pip-audit, dependency_check
are documented examples). Public API functions:
- `get_tools_for_phase(phase)` — filter tools by phase
- `get_tool(name)` — single tool lookup
- `is_tool_available(name)` — availability check
- `evaluate_gate(tool_name, context)` — gating logic

### Layer 4: Consumer (`tool_core/registry.py`)

PATH-scanning registry that imports `TOOLS` from `tool_definitions.py` and
checks which tools are installed on the system. It does NOT define its own
tool registry — it reads from Layer 3.

## TS-Side Registry

The docstring references `Argus-Tui/packages/opencode/src/argus/workflows/tool-registry.ts`
as the TS-side independent registry for planning metadata (capabilities, scoring,
auth gating). **This file does not exist in this checkout.** No YAML files
exist in the `Argus-Tui` directory either. The TS-side registry may be from a
different branch or may not have been implemented in this version.

## Conclusion: NOT a Duplicate Registry Problem

| Concern | Finding |
|---|---|
| Are tool_definitions.py and _generated_tools.py duplicates? | **No.** `_generated_tools.py` is auto-generated from YAML; `tool_definitions.py` imports it and adds overrides. Deliberate layered architecture. |
| Are tool_definitions.py and tools/definitions/*.yaml duplicates? | **No.** YAML is the source; Python is the compiled output + overrides. |
| Is tool_core/registry.py a competing registry? | **No.** It's a PATH-scanning consumer that reads from tool_definitions. |
| Is the TS tool-registry.ts a drift risk? | **File doesn't exist in this checkout.** No actionable concern. |

## Recommendation

No action needed on the registry itself. However:

1. **Keep the pipeline documented.** The layered design is correct but not obvious
   to new contributors. The docstring in `tool_definitions.py` already explains it
   well — ensure it stays up-to-date.
2. **If TS-side tool-registry.ts is ever implemented**, it should explicitly
   document that it's independent from the Python side (different schema,
   different purpose), as the current docstring states.
3. **Script automation:** The `generate_tool_defs.py` regeneration step should be
   part of CI to ensure `_generated_tools.py` never drifts from YAML sources.
