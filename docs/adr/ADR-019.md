# ADR-019: File Cache Fallback — Rejected

**Status:** Accepted

**Date:** 2026-06-09

## Context

During the Phase 2 cache controls design, local JSON file cache fallback was considered as a secondary storage layer when Redis is unavailable. The rationale was: if Redis goes down, the cache could fall back to local disk storage so the scan pipeline still benefits from cached tool results.

The current `WorkerCache` already degrades gracefully when Redis is unavailable:

- `WorkerCache.get()` returns `None` (cache miss)
- `WorkerCache.set()` silently handles failures (logged, not raised)

Redis is a performance optimization, not an availability dependency. Every tool invocation path already handles the "no cache" case correctly.

## Decision

**Do not implement local JSON file cache fallback.**

The security and maintenance costs outweigh the marginal benefit.

### Security

A file cache introduces several attack surfaces that do not exist with Redis:

1. **Fernet encryption required:** Cache data may contain findings data, target URLs, and tool output. If stored on disk without encryption, any process with filesystem access can read cached scans. Fernet (symmetric encryption) introduces key lifecycle management — where is the key stored? How is it rotated? What happens if the key is lost?

2. **Namespace-based sensitivity classification:** To determine whether a cache entry needs encryption, every write site must classify its data as sensitive/non-sensitive. This means `WorkerCache.set()` (currently a simple key-value write) must be aware of data sensitivity, coupling storage with a cross-cutting security concern.

3. **Path traversal prevention:** File cache requires constructing filesystem paths from cache keys. Malicious or malformed keys could escape the cache directory via path traversal (`../../etc/passwd`). Redis handles this naturally — key name and storage location are independent.

4. **File permissions:** The cache directory must be readable by the worker process but not by unauthorized users. On multi-tenant systems, this requires careful `umask` and directory permission management that Redis handles transparently.

### Maintenance

5. **Two code paths to maintain:** Every cache operation would need an in-memory-or-Redis primary path and a fallback-to-disk path. This doubles the testing surface for a feature that would rarely be exercised.

6. **Staleness without TTL:** File cache entries persist indefinitely unless explicitly cleaned up. Unlike Redis, there's no built-in TTL expiry. A garbage collection mechanism would be needed, adding another maintenance burden.

### Marginal Benefit

7. **Redis unavailability is rare and fast to recover:** Redis is a local process in most Argus deployments. Connection failures are typically transient (seconds). The cache miss cost on a single tool invocation (< 1 second) is less than the I/O cost of reading from disk.

8. **Worst case is a cache miss:** When Redis is down, every tool invocation re-executes. This is the same behavior as `--no-cache` mode. The pipeline still produces correct results — just slightly slower.

## Consequences

- Simpler codebase — no Fernet encryption, no key management, no path traversal prevention, no garbage collection
- `WorkerCache` remains a thin Redis wrapper with graceful degradation
- When Redis is unavailable, the system degrades to "always fetch fresh" — correct but slower
- Security posture is improved — no sensitive cache data on disk, no encryption key to manage

## When to Revisit

This decision should be revisited only if production evidence shows:

1. **Redis is remote and frequently unavailable** — e.g., a cloud deployment where Redis is a separate service with intermittent connectivity
2. **Certain cache data is extremely expensive to regenerate** — e.g., >30 seconds per lookup, making repeated cache misses painful
3. **Argus needs an offline single-user mode without Redis** — a deployment model where Redis is entirely absent

None of these conditions are met today. If they arise, the file cache design must address all four security concerns (encryption, classification, path traversal, permissions) before implementation.

## Alternatives Considered

- **SQLite-based file cache:** Adds a build dependency on SQLite and introduces the same security concerns (encryption, path traversal). Rejected for the same reasons.

- **Memory-only fallback (in-process dict):** Simpler than file cache but loses data on process restart. Provides no benefit over Redis's built-in in-memory caching. Rejected as redundant.

- **Status quo (graceful None return):** The current behavior is already acceptable — Redis failures produce cache misses, not crashes. This is the selected approach.
