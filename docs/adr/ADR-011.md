# ADR-011: Credential Storage Evolution

**Status:** Accepted

**Date:** 2026-06-02

## Context

The `--creds` JSON file approach (ADR-004) stores credentials in plaintext on disk. While acceptable for v5, this is not a long-term solution. A documented evolution path is needed so users understand how credential security will improve without holding up the v5 release.

## Decision

Use `--creds` JSON file for v5. The `CredentialStore` is defined as an abstract interface (or Protocol) with a single implementation (`JsonFileCredentialStore`). Document the future evolution path in the architecture docs: OS keychain integration (macOS Keychain, libsecret for Linux, Windows Credential Manager) for v6, HashiCorp Vault integration for enterprise deployments, and an encrypted secrets store as an intermediate step. The interface supports swapping backends without changing any caller code. A `credentialStoreVault` document in the design directory captures these plans.

## Consequences

- v5 ships on time with the simple JSON file approach
- Future backend swaps require no changes to any code that consumes credentials — only the store implementation changes
- Documented evolution path manages user expectations about security improvements
- Maintaining the interface abstraction adds a small indirection layer in v5 that is not yet exercised by multiple implementations
- The interface may need revision when OS keychain integration reveals practical constraints not anticipated in the v5 design

## Alternatives Considered

- **Full keychain integration in v5:** Increases scope for the initial release, adds platform-specific testing burden, and delays shipping. Better suited as a focused v6 feature.
- **Plaintext as permanent solution:** Unacceptable security risk — credentials remain on disk indefinitely in readable form, and users cannot safely share machines or commit config to version control.
