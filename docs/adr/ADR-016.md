# ADR-016: LLM Availability & Fallback Strategy

**Status:** Accepted

**Date:** 2026-06-02

## Context

LLM providers can fail for many reasons (network outage, rate limiting, service degradation, authentication errors). The system must degrade gracefully when the LLM is unavailable, continue assessments with deterministic fallback, and recover automatically when the LLM becomes available again. A brittle "fail on first error" approach would make the tool unreliable.

## Decision

Implement a layered detection and fallback system:
1. **Provider layer** detects raw failures (timeout, HTTP error, authentication failure)
2. **Bridge layer** normalizes all provider errors into a single `LLMUnavailableError` type and implements a circuit breaker that tracks failure counts
3. **Planner layer** catches `LLMUnavailableError` at phase boundaries and enters per-phase deterministic fallback (see ADR-003)

After 3 LLM failures within a 10-minute sliding window, the system enters a sticky `DEGRADED` mode where LLM calls are skipped entirely. Recovery probes (simple prompt every 60 seconds) re-enable the LLM after a successful response, exiting `DEGRADED` mode. Failures during `DEGRADED` mode are silently skipped without incrementing the counter.

## Consequences

- Graceful degradation — the system continues operating during LLM outages, using deterministic fallback for planning
- Circuit breaker prevents wasted API calls during extended outages — no point hammering a down service
- Automatic recovery probes restore LLM functionality without user intervention
- Sticky `DEGRADED` mode prevents rapid toggling between LLM and deterministic modes during intermittent failures
- Per-phase fallback means some phases may use LLM while others use deterministic — consistency across phases is not guaranteed
- Recovery probes add minimal latency (one probe every 60 seconds) and negligible cost

## Alternatives Considered

- **Fail on first LLM error:** Too brittle — a transient network glitch would abort the entire assessment, producing incomplete results and frustrating users.
- **Always-retry without circuit breaker:** Wastes time and API quota retrying a service that is clearly down. Users wait for multiple timeouts before the system gives up.
- **No fallback (LLM required):** Makes offline operation impossible — the tool is unusable without network access, contradicting a core requirement for v5.
