# ADR-006: Backend Connection Pooling

**Status:** Accepted

**Date:** 2026-06-02

## Context

The Python backend uses psycopg2 for PostgreSQL database access. Multiple concurrent requests can arrive during assessment execution, and establishing a new connection per request is expensive and wasteful under load. A connection pooling strategy is needed to reuse connections efficiently.

## Decision

Use a single `psycopg2.pool.ThreadedConnectionPool` instance at the application level. Remove the raw connection-string path entirely — every database operation goes through the pool. The `_release_connection()` method returns connections to the pool after use; it never closes them. The pool size is configurable via environment variable with a default of `min=2, max=10`. Connections are lazily created up to `max`, and idle connections beyond `min` are pruned after a configurable timeout.

## Consequences

- Eliminates connection establishment overhead for burst workloads
- Predictable database load — maximum concurrent connections capped by pool size
- Single pool simplifies lifecycle management (initialize on startup, close on shutdown)
- Connections never closed during normal operation — long-running assessments keep connections open throughout
- Pool exhaustion under extreme load blocks callers until a connection is released
- Thread safety must be verified — `ThreadedConnectionPool` is thread-safe but callers must not share connections across threads

## Alternatives Considered

- **Per-request connections:** Creates and destroys a connection for every request. Under assessment load this leaks file descriptors and increases latency by ~50ms per connection handshake.
- **Connection-per-thread:** Wastes resources when threads are idling during I/O-bound assessment phases. A pooled approach shares connections more efficiently across burst workloads.
