# ADR-006: Backend Connection Pooling

**Status:** Accepted

**Date:** 2026-06-02 (Revised 2026-06-09)

## Context

The Python backend uses psycopg2 for PostgreSQL database access. Multiple concurrent requests can arrive during assessment execution, and establishing a new connection per request is expensive and wasteful under load. A connection pooling strategy is needed to reuse connections efficiently.

## Decision

Use a single `psycopg2.pool.ThreadedConnectionPool` instance managed by the `ConnectionManager` singleton in `database/connection.py`. The pool is lazily initialized on first use via `_ensure_pool()` and is thread-safe with double-checked locking.

```python
self._pool = pool.ThreadedConnectionPool(min_connections, max_connections, conn_string)
```

### Pool Configuration

Pool size is configured via environment variables:

| Variable | Default | Description |
|----------|---------|-------------|
| `DB_POOL_MIN` | `2` | Minimum persistent connections |
| `DB_POOL_MAX` | `20` | Maximum connections |
| `DB_STATEMENT_TIMEOUT_MS` | `30000` | Per-query timeout (30s) |
| `DB_SLOW_QUERY_MS` | `500` | Threshold for slow-query logging |
| `DB_SSLMODE` | `prefer` | SSL mode for connections |
| `PGBOUNCER_MODE` | `session` | PgBouncer compatibility mode |

### Connection Lifecycle

1. **Acquisition:** `get_connection()` acquires a connection from the pool with retry + exponential backoff + uniform jitter via `threading.Condition.wait()` to avoid busy-wait under contention
2. **Statement timeout:** Every connection gets `SET statement_timeout` enforced on checkout to prevent runaway queries from blocking the pool
3. **Usage:** Callers use the `connection()` or `cursor()` context managers which auto-commit and release
4. **Release:** `release_connection()` returns the connection to the pool via `pool.putconn()` and notifies any waiting threads

### Remaining Raw-Connection Paths

The ADR's goal to "remove the raw connection-string path entirely" is **partially fulfilled**:

- **`database/connection.py` `connect()` function** — A convenience helper that creates one-off connections via `psycopg2.connect()`. Used when the pool is not available or for one-off operations.
- **`tasks/bugbounty.py`** — Still uses `psycopg2.connect()` directly in some code paths.
- **`state_machine.py`** — Emits a deprecation warning when called with a raw connection string: *"connection pooling is preferred. Remove this parameter."*

All new code should use the shared pool via `get_db().get_connection()` or the `connection()`/`cursor()` context managers.

## Consequences

- Eliminates connection establishment overhead for burst workloads
- Predictable database load — maximum concurrent connections capped by pool size
- Single pool simplifies lifecycle management (initialize on startup, close on shutdown)
- Connections never closed during normal operation — long-running assessments keep connections open throughout
- Pool exhaustion under extreme load blocks callers until a connection is released (with configurable timeout, default 30s)
- Thread safety via `ThreadedConnectionPool` — callers must not share connections across threads
- Statement timeout prevents runaway queries from monopolizing pool connections
- Condition-variable wait replaces busy-polling for connection acquisition (fixes B.03)

## Alternatives Considered

- **Per-request connections:** Creates and destroys a connection for every request. Under assessment load this leaks file descriptors and increases latency by ~50ms per connection handshake.
- **Connection-per-thread:** Wastes resources when threads are idling during I/O-bound assessment phases. A pooled approach shares connections more efficiently across burst workloads.
