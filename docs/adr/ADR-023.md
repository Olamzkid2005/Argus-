# ADR-023: Tool Definition Boundaries

**Status:** Accepted

**Date:** 2026-06-09

## Context

The Argus codebase has two YAML schemas for tool definitions:

| Concern | Python YAML (`tools/definitions/*.yaml`) | TUI YAML (`tool-definitions.yaml`) |
|---------|----------------------------------------|-------------------------------------|
| Domain | **Execution** — how to run a tool | **Workflow** — when to use a tool |
| Content | `command`, `args`, `parameters`, `timeout`, `risk_level`, `signal_quality` | `consumes`, `provides`, `supports_api`, `min_version`, `scoring.confidence_score` |
| Tool set | 65 definitions (including internal pipeline components) | Curated subset of user-facing security scanners |
| Ownership | Python workers team | TypeScript TUI planner team |

The initial assumption was that the Python YAML should be the single source of truth, and the TUI YAML should be generated from it. This assumption was incorrect — the two schemas model fundamentally different domains and share only `name` and `capabilities` as overlapping fields.

Attempting to generate one from the other would produce incomplete definitions or require enriching 65 YAMLs with fields from a different domain (e.g., adding `consumes`/`provides` to every Python YAML, or adding `command`/`args` to every TUI YAML).

## Decision

**Both YAMLs are authoritative in their own domain. Neither is generated from the other.**

The overlapping fields that SHOULD agree across both schemas are:
- Tool existence (tools present in both where expected)
- Capability names (`capabilities` in Python YAML; `capabilities` in TUI YAML)
- Risk level classification (`risk_level` in Python YAML; `destructive` in TUI YAML)

A validation script (`validate_tool_alignment.py`) checks these overlapping fields for consistency. It does NOT validate domain-specific fields:
- `consumes` / `provides` (workflow-only — not in Python YAML)
- `command` / `args` / `parameters` (execution-only — not in TUI YAML)
- `min_version` / `version_cmd` (workflow-only)
- `scoring.confidence_score` (workflow-only) vs `signal_quality` (execution-only)

The validation script runs in CI via `python3 scripts/validate_tool_alignment.py --check`.

## Consequences

- Adding a new tool requires updating both YAMLs independently — there is no single "register tool here" step.
- The validation script catches drift early (CI fails if overlapping fields diverge).
- Each schema evolves independently within its own domain. Adding a new workflow field (e.g., `requires_approval`) does not require changes to 65 Python YAMLs.
- The validation is intentionally lenient — mismatches where Python has capabilities the TUI doesn't know about are errors, but mismatches the other direction are warnings (the TUI can be more abstract).

## Alternatives Considered

- **Generate TUI YAML from Python YAML:** Would require enriching all 65 Python YAMLs with workflow-specific fields like `consumes`, `provides`, and `scoring`. These fields don't belong in the execution schema. The generator would produce incomplete definitions or require a second enrichment step. Rejected.
- **Generate Python YAML from TUI YAML:** The TUI YAML has ~42 tools (user-facing scanners only), while Python has ~65 (including internal pipeline components like `assessment_orchestrator`, `finding_correlation_engine`). A generator would miss 23 internal tools. Rejected.
- **Single unified YAML format:** Would require a superset schema that includes both execution and workflow fields. Every tool definition would need both sets of fields, bloating the execution-only definitions with workflow metadata they never use. Rejected.
