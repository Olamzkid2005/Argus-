# ADR-005: Error Recovery Policy

**Status:** Accepted

**Date:** 2026-06-02

## Context

During an assessment, individual tools and phases may fail for various reasons — network timeouts, service unavailability, transient errors, or destructive action safeguards. A consistent error recovery policy is needed to determine whether execution should retry, skip, or abort, and this policy must be configurable per phase and per tool.

## Decision

Define a per-phase `ErrorRecovery` matrix with three strategies: `retry_once_then_skip`, `skip_and_continue`, and `fail_fast`. The matrix is defined in YAML within each phase definition. Tool properties can override the phase-level default (e.g., tools marked `destructive: true` default to `fail_fast` regardless of the phase setting). The error handler evaluates the strategy, executes the appropriate action (retry with backoff, skip phase/tool, or abort the assessment), and records the outcome in the evidence store.

## Consequences

- Fine-grained control — sensitive phases can use `fail_fast` while reconnaissance phases use `skip_and_continue`
- YAML-defined policies are user-visible and overridable in project config
- `retry_once_then_skip` handles transient failures without infinite retry loops
- Matrix complexity grows with the number of phases — each new phase must define its policy
- Override by tool property adds an implicit coupling between the tool definition and the recovery system

## Alternatives Considered

- **Global retry policy:** Too rigid — a reconnaissance tool timeout and a exploitation tool failure need different responses, but a global policy treats them identically.
- **Always-fail:** Aborts the entire assessment on any minor issue (e.g., a single port scan timeout), producing incomplete results and frustrating users.
