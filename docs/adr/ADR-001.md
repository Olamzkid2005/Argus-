# ADR-001: MCP Transport Layer

**Status:** Accepted

**Date:** 2026-06-02

## Context

Argus requires a communication mechanism between the orchestrator and individual tool servers (Model Context Protocol servers). The transport layer must support spawning subprocesses, exchanging structured messages, and handling errors gracefully across platforms (macOS, Linux, Windows). The transport choice affects latency, portability, and operational complexity.

## Decision

Use stdio-based JSON-RPC 2.0 subprocess communication. The `mcp_transport.py` module spawns `python3 mcp_server.py` as a child process. Requests and responses are exchanged as JSON-RPC 2.0 messages over the subprocess's stdin/stdout streams. Each message is a single JSON object terminated by a newline. The transport layer manages process lifecycle (spawn, health-check, graceful shutdown, force-kill on timeout).

## Consequences

- Simple implementation — no network ports, no daemon management, no serialization framework beyond Python's `json` module
- Platform-independent by construction — stdin/stdout work identically on all operating systems
- Process isolation — a crashed MCP server does not take down the orchestrator
- Subprocess startup latency on each invocation — mitigated by keeping long-lived server processes
- No support for remote MCP servers — stdio inherently requires co-located processes

## Alternatives Considered

- **TCP daemon:** Requires port management, firewall configuration, and port-conflict resolution. Adds operational surface area for a v5 CLI tool.
- **Unix socket:** Faster than TCP loopback but has cross-platform issues — Windows named pipes have different semantics, making portable code more complex.
