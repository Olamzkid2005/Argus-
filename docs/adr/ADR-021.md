# ADR-021: Streaming Error Events

**Status:** Accepted

**Date:** 2026-06-09

## Context

The streaming system (`streaming.py`) uses a Port/Adapter pattern with `StreamManager` (singleton), `Event` dataclass, and 18 existing `emit_*` convenience functions (`emit_tool_start`, `emit_error`, `emit_finding`, etc.). When error hints were introduced in Phase 1, we needed a way to deliver them to the TUI.

Two questions arose:
1. Should error hints be delivered as a new event type, or overload the existing `ERROR` event?
2. Should error hints participate in the transactional event stream (persist → commit → emit ordering)?

The `ERROR` event currently carries raw error text (`{"error": "...", "phase": "..."}`). Overloading it with structured hint data would break existing consumers that expect simple error strings. The transactional stream (`SafeEventEmitter`) is designed for events that must be persisted before emission — error hints are UI-only guidance that should not be queued behind a DB commit.

## Decision

**`ERROR_HINT` is a separate event type** from `ERROR`. They are independently consumable by the TUI on separate event channels.

```python
class EventType:
    ERROR = "error"
    ERROR_HINT = "error_hint"  # NEW
```

The `emit_error_hint()` convenience function follows the existing `emit_*` pattern:

```python
def emit_error_hint(engagement_id: str, hint: ErrorHint) -> None:
    get_stream_manager().publish(Event(
        type=EventType.ERROR_HINT,
        engagement_id=engagement_id,
        data=hint.to_dict(),
    ))
```

Key rules:
1. **Error hints do NOT participate in the transactional event stream.** They bypass `_maybe_transactional()` and publish directly. Error hints are UI ephemera, not audit-log data.
2. **The hint includes `tool` and `error_id` metadata** so the TUI can associate hints with specific tool failures.
3. **The `ERROR_HINT` event carries a structured `ErrorHint` dict** (summary, detail, remediation, hint_command, docs_url) — not raw error strings.

## Consequences

- TUI can render error hints differently from errors — toasts vs inline messages, different iconography, actionable buttons for remediation.
- Existing `ERROR` event consumers are unaffected — no breaking schema changes.
- Error hints bypass transaction ordering, so they appear immediately even if the DB commit is slow.
- Consumers must handle two independent event types instead of one, but this is offset by simpler per-type rendering logic.

## Alternatives Considered

- **Overload ERROR event with optional hint fields:** Would require consumers to check for hint fields on every error event, complicating the happy path. Schema migration risk — old consumers would ignore hint fields silently. Rejected.
- **Add hint fields to existing ToolComplete event:** ToolComplete conveys "a tool finished running" — error hints are about "a tool failed and here's what to do". Different semantics. Rejected.
- **Deliver hints via the transactional stream:** Would delay hints behind DB commits, making them appear late or not at all if the commit fails. Error hints are most valuable immediately after the error occurs. Rejected.
