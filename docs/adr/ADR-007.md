# ADR-007: Threat Intel Caching

**Status:** Accepted

**Date:** 2026-06-02

## Context

Argus queries external threat intelligence sources for CVE details and EPSS scores during assessments. These API calls are rate-limited, add latency, and may fail under load or without network access. A caching layer is needed to avoid redundant API calls and improve assessment speed.

## Decision

Use a two-tier caching strategy: Redis-backed TTL cache (1-hour TTL) as the primary layer, with an in-memory `dict` fallback when Redis is unavailable. Cache keys follow the format `threat_intel:cve:{CVE_ID}`. On a cache miss, the system queries the upstream API, stores the result in both Redis and the in-memory fallback, and returns it. The in-memory fallback is process-scoped and is lost on restart but provides resilience when Redis is down. The TTL is configurable via project config.

## Consequences

- Eliminates redundant API calls — subsequent lookups of the same CVE within the TTL window hit cache
- Graceful degradation — in-memory fallback keeps assessments running when Redis is unavailable
- 1-hour TTL balances freshness with cache-hit rate for CVE data that changes infrequently
- In-memory cache is lost on process restart — transient cache misses may occur after restart
- Redis adds an infrastructure dependency — users must have Redis available or accept the in-memory-only mode

## Alternatives Considered

- **No cache:** Repeated lookups of the same CVE across multiple targets cause redundant API calls, hitting rate limits and slowing assessments under load.
- **Only in-memory cache:** Lost on every restart, causing a burst of API calls after each restart. Redis persistence provides continuity across assessment sessions.
