# ADR-007: Threat Intel Caching

**Status:** Accepted

**Date:** 2026-06-02 (Revised 2026-06-09)

## Context

Argus queries external threat intelligence sources for CVE details and EPSS scores during assessments. These API calls are rate-limited, add latency, and may fail under load or without network access. A caching layer is needed to avoid redundant API calls and improve assessment speed.

## Decision

Use a two-tier caching strategy implemented in the `IntelligenceEngine` class (`intelligence_engine.py`):

1. **Primary tier:** Redis-backed cache via the shared `WorkerCache` (`cache.py`)
2. **Fallback tier:** In-memory `dict` (`_in_memory_cache: dict[str, tuple[float, Any]]`) gated by expiry timestamps

### Cache Architecture

```python
def _cache_get(self, key: str) -> Any | None:
    # 1. Try Redis first
    cached = cache.get(key)
    if cached is not None:
        return cached
    # 2. Fall back to in-memory dict
    entry = self._in_memory_cache.get(key)
    if entry is not None:
        expiry, value = entry
        if time.time() < expiry:
            return value
        del self._in_memory_cache[key]  # purge expired
    return None

def _cache_set(self, key: str, value: Any, ttl: int | None = None) -> None:
    ttl = ttl or THREAT_INTEL_CACHE_TTL  # default 3600s (1 hour)
    # Always write to both tiers
    cache.set(key, value, ttl)                          # Redis (best-effort)
    self._in_memory_cache[key] = (time.time() + ttl, value)  # in-memory
```

### Cache Keys

| Pattern | Content | TTL |
|---------|---------|-----|
| `threat_intel:nvd:{sorted_cve_ids}` | NVD CVE details (CVSS score, description, references) | 1 hour |
| `threat_intel:epss:{sorted_cve_ids}` | EPSS exploitability scores per CVE | 1 hour |

The 1-hour TTL aligns with NVD/EPSS dataset update cycles (every 2-8 hours).

### Behavior

- **Redis available:** Read/write through Redis. In-memory cache is written but unused.
- **Redis unavailable:** `cache.get()` returns `None` gracefully → falls through to in-memory cache → hit or miss.
- **Cache miss:** System queries the upstream NVD/EPSS API, stores the result in both tiers, and returns it.
- **Process restart:** In-memory cache is lost. Redis persistence provides continuity across runs.

## Consequences

- Eliminates redundant API calls — subsequent lookups of the same CVE within the TTL window hit cache
- Two-tier fallback keeps assessments running when Redis is unavailable (in-memory cache persists within process lifetime)
- 1-hour TTL balances freshness with cache-hit rate for CVE data that changes infrequently
- In-memory cache is process-scoped — lost on restart, but Redis provides cross-session persistence
- Best-effort Redis writes — `cache.set()` logs failures but never crashes the enrichment pipeline
- Cache keys include sorted CVE IDs so the same CVE set always produces the same key, maximizing hit rate

## Alternatives Considered

- **No cache:** Repeated lookups of the same CVE across multiple targets cause redundant API calls, hitting rate limits and slowing assessments under load.
- **Only in-memory cache:** Lost on every restart, causing a burst of API calls after each restart. Redis persistence provides continuity across assessment sessions.
- **Only Redis cache:** While Redis is the primary tier, the in-memory fallback prevents a full recalculation on every cache miss when Redis is down. A single missing tier should not force an API call.
