# ADR-012: Assessment Execution Model

**Status:** Accepted

**Date:** 2026-06-02

## Context

Assessments consist of multiple phases, each containing multiple tools. The execution model determines whether phases run sequentially or concurrently, and whether tools within a phase run in series or parallel. The right balance between speed and complexity must be chosen for v5.

## Decision

Use sequential workflow execution for v5: phases execute in order, one after another. Within a phase, tools run concurrently when the phase definition includes `execution: parallel`; otherwise they run sequentially in definition order. Parallel DAG-style execution (where phases can run concurrently based on dependency resolution) is reserved for v5.1. The `PhaseExecutor` interface supports this evolution by accepting a `PhaseExecutionRequest` that can express dependency information without changing the executor contract.

## Consequences

- Predictable execution order — assessment results are reproducible because phases always run in the same sequence
- Intra-phase parallelism improves speed for independent tools (e.g., concurrent port scans)
- Simple state management — no need for dependency graph resolution or shared-state synchronization between phases
- Sequential phases are slower for tasks that could run in parallel (e.g., recon while enumeration runs)
- Migration to parallel DAG in v5.1 will require updating phase definitions to declare dependencies

## Alternatives Considered

- **Full parallel DAG in v5:** Introduces significant complexity in state management, error handling, and dependency resolution. The benefit is marginal for most v5 assessments where phases have natural sequential dependencies (recon before exploitation).
- **Fully sequential (no parallel tools):** Too slow — waiting for each tool to complete before starting the next one, even when tools are independent, makes assessments unnecessarily time-consuming.
