# ADR-002: Evidence Storage

**Status:** Accepted

**Date:** 2026-06-02

## Context

Argus needs to persist assessment evidence including metadata (engagement records, findings, evidence packages, artifacts) and potentially large binary blobs (screenshots, HAR files, network logs). The storage backend must work offline, be safe for concurrent access, and require zero infrastructure setup from the user.

## Decision

Use a hybrid storage model: SQLite (via Drizzle ORM, WAL journal mode) for all metadata, and the filesystem for binary blobs organized in per-package directories. Each evidence package directory contains a `manifest.json` that lists artifacts and their hashes for integrity verification. Drizzle ORM provides type-safe queries and migration management without the overhead of a full PostgreSQL setup. WAL mode allows concurrent readers without write-lock contention.

## Consequences

- Zero-infrastructure setup — SQLite ships with Python's stdlib, no database server needed
- Type-safe queries via Drizzle ORM schema definitions
- WAL mode enables concurrent reads during evidence ingestion
- Binary blobs remain directly accessible on disk for external tooling
- Filesystem layout must be consistent across platforms — handled by `pathlib` for OS-agnostic paths
- No built-in replication or remote access — evidence is local-only by design

## Alternatives Considered

- **Pure filesystem with JSON indexes:** Concurrent writes from parallel tool executions would cause race conditions and index corruption without a proper transactional layer.
- **Pure PostgreSQL:** Too heavy for a CLI tool — requires a running database server, connection configuration, and user setup that contradicts the offline/no-infra requirement.
- **S3-compatible storage:** Requires network connectivity and credentials, violating the offline-capable requirement for v5.
