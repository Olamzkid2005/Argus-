# ADR-024: Report Export Architecture

**Status:** Accepted

**Date:** 2026-06-09

## Context

Argus generates executive security reports through `ExecutiveReportGenerator` (a pipeline tool). It produces structured data (dict with severity breakdown, executive summary, top findings, markdown). Users needed a standalone HTML report format that could be shared via email or saved to disk.

Three approaches were considered:
1. Add HTML rendering + file I/O directly to `ExecutiveReportGenerator`
2. Create a separate renderer library that pipeline tools can optionally call
3. Create separate renderers at the application boundary, keeping pipeline tools pure

Approach 1 would couple rendering logic with pipeline execution. Approach 2 would make pipeline tools impure (they'd need to write files or launch browsers). Approach 3 keeps the pipeline portable across deployment contexts (local CLI, background workers, Docker, remote MCP).

## Decision

**Rendering is a pure function at the library layer. File I/O and browser launching live at the CLI/application boundary.**

```
ExecutiveReportGenerator (pipeline tool)
      │
      └── structured report dict
            ├── severity_breakdown
            ├── executive_summary
            └── markdown (backward compat)
      │
      ▼
Report Store (persisted structured data)
      │
      ├──────────────────────────┬──────────────────────┐
      ▼                          ▼                      ▼
Markdown Renderer           HTML Renderer          JSON Renderer (future)
(reporting/, pure)          (reporting/html_report.py, pure)
      │                          │                      │
      ▼                          ▼                      ▼
exporter.py (I/O boundary)  exporter.py (I/O)      exporter.py (I/O)
      │                          │                      │
      ▼                          ▼                      ▼
 report.md                  report.html            report.json
```

Key rules:
1. **`ExecutiveReportGenerator` produces pure data** — no file I/O, no browser launches, no side effects.
2. **HTML rendering is a pure function** — `render_html_report(report_data) → str`. Zero external dependencies (no CSS CDN, no JS CDN, no fonts).
3. **File writing is an application-layer concern** — `exporter.py` is the only module that calls `Path.write_text()`.
4. **Browser launching is a CLI-only concern** — `--open` flag exists only in the CLI arg parser, never in library code.
5. **XSS safety** — all user data is passed through `html.escape()` before embedding. Data in `<script>` blocks uses `json.dumps()`, not string interpolation.

## Consequences

- Pipeline tools remain pure and testable — no filesystem mocking needed in unit tests.
- HTML renderer can be tested by asserting on the returned string — no temp files, no browser automation.
- Same renderer works in local CLI, background workers, Docker containers, and remote MCP workers — no display dependency.
- The `--open` flag is trivially testable (it just runs `webbrowser.open()`) and independent of the render quality.
- Adding a new format (JSON, PDF) requires: (a) a new pure renderer in `reporting/`, (b) a new format case in `exporter.py`. Two independent, testable changes.

## Alternatives Considered

- **Embed HTML rendering in ExecutiveReportGenerator:** Would couple I/O with pipeline execution. The report generator would need to know about file paths, browser availability, and export formats. Not portable across deployment contexts. Rejected.
- **Embed HTML rendering as a pipeline step:** Would add side effects to the pipeline (file writes). A pipeline step should produce data, not write files. Rejected.
- **Use an external templating engine (Jinja2):** Jinja2 is already a dependency (used for compliance reports), but introducing it for HTML reports creates a new templating surface. Python f-strings and string concatenation are sufficient for a single self-contained HTML file. Rejected for simplicity.
- **Ship report as a React SPA:** Overkill for a single-file report. Requires npm dependencies, a build step, and adds ~50KB of framework boilerplate. Rejected.
