# ADR-013: Capability Taxonomy

**Status:** Accepted

**Date:** 2026-06-02

## Context

Argus tools declare their capabilities so the Planner can select appropriate tools for each assessment target. A consistent, validated taxonomy is needed to prevent silent drift where tools claim capabilities that the Planner does not recognize or that conflict with YAML definitions.

## Decision

Define a canonical `Capability` enum in `capabilities.ts` as the single source of truth for all capability values. Every capability value that appears in YAML phase definitions is validated against this enum at load time. Capability values are always enum members, never free-form strings. Adding a new capability requires:
1. Adding the value to the `Capability` enum (TypeScript change)
2. Updating YAML definitions that reference the new capability
This dual-touch requirement ensures that capability changes are intentional and reviewed.

## Consequences

- Type-safe capability matching — the Planner can switch over enum variants exhaustively
- YAML validation catches typos and unknown capabilities at config load time, not during execution
- No silent drift — a capability cannot be used in YAML unless it exists in the enum
- Adding a capability requires touching both TypeScript and YAML — this is intentional, not a bug
- Enum must be kept in sync with the YAML schema documentation — potential for docs to lag behind code

## Alternatives Considered

- **String-based capabilities:** No validation — a typo in YAML (`reconnaissance` vs `reconnaisance`) silently produces an empty tool set, and the Planner matches nothing. Debugging requires comparing strings at runtime.
- **JSON Schema for capabilities:** Requires maintaining a separate schema file that duplicates the enum definition. The schema and enum would drift apart without additional tooling to enforce consistency.
