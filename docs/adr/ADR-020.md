# ADR-020: Error Hint Architecture

**Status:** Accepted

**Date:** 2026-06-09

## Context

When a security tool fails during a scan (SSL error, rate limit, tool not found), the user sees a raw error message or stack trace. These messages are technically accurate but operationally useless for most users — they don't explain *what to do about it*.

The existing `error_classifier.py` (529 lines) already classifies errors into `ErrorCategory`, `ErrorSeverity`, and `ErrorCode` for *operational* purposes (retry logic, alerting). This classification is deterministic and code-based, but it produces no user-facing output — only internal metadata like `should_retry` and `retry_delay_seconds`.

Two approaches were considered:
1. Embed user-facing text inside `error_classifier.py` alongside the operational classification
2. Create a separate presentation layer that consumes the classification

Approach 1 would couple operational routing with user-facing copy, making it impossible to change error messages without touching retry logic.

## Decision

Create a **separate presentation layer** — `utils/error_hints.py` — that consumes `CodeBasedClassification` from `error_classifier.py` and translates it to user-facing `ErrorHint` objects.

The architectural boundary is strict:

```
Raw Exception
      │
      ▼
error_classifier.py      ← Operational: classification, retry, alerting
      │
      ├── ErrorCode (27 codes — deterministic)
      ├── ErrorCategory (10 categories — pattern fallback)
      └── CodeBasedClassification
      │
      ▼
error_hints.py            ← Presentation: user-facing text only
      │
      └── ErrorHint
             ├── summary
             ├── detail
             ├── remediation
             ├── hint_command
             └── docs_url
      │
      ▼
streaming.py (emit_error_hint) → TUI
```

Key rules:
1. **`error_hints.py` must NOT re-implement detection logic** — no `is_ssl_cert_error()`, no `is_rate_limit_error()`. It receives classification from `error_classifier.py` and maps it to text.
2. **Three-tier priority**: ErrorCode → ErrorCategory → generic fallback. The most specific match wins.
3. **The ONLY exception** is tool-specific remediation (nuclei templates missing, nmap permissions, sqlmap connection errors). This inspects stderr for *remediation guidance*, not re-classification.
4. **Hint generation must never crash the tool** — wrapped in its own `try/except` at every call site.

## Consequences

- Error messages can be improved independently of classification logic — changes to `error_hints.py` never affect retry behavior.
- Adding a new error type requires: (a) adding an `ErrorCode` in `error_classifier.py` for deterministic classification, (b) adding a hint mapping in `error_hints.py`. Two independent, testable changes.
- 19 ErrorCode-specific hints + 10 ErrorCategory fallbacks + tool-specific patterns for 5 tools — covers the top-10 failure scenarios.
- Tool-specific stderr patterns are maintained separately from classification patterns, preventing the two concerns from entangling.

## Alternatives Considered

- **Embed hints in error_classifier.py:** Would couple presentation with operational routing. Changing error text would risk changing retry behavior. Rejected.
- **Single error_hints.py with duplicate classification logic:** Would create two classification systems that could disagree. When the error hints said "rate limit" but the operational classifier said "transient", there would be no single source of truth. Rejected.
- **No user-facing hints (status quo):** Users see stack traces and raw error messages. Rejected because it degrades the CLI experience for non-expert users.
