# ADR-009: Feature Flag Strategy

**Status:** Accepted

**Date:** 2026-06-02

## Context

v5 introduces several new modules (browser verification, workflow registry, engagement store) that are not yet stable or may not be needed by all users. A mechanism is needed to gate these features so v5 behaves identically to v4 by default while allowing early adopters to opt in.

## Decision

Use per-module activation flags in the project configuration file. Each flaggable module (`browser_verification`, `workflow_registry`, `engagement_store`) has a boolean `enabled` key under its configuration section. All modules default to `off` in v5. CLI flags (`--enable-browser-verification`) and environment variables (`ARGUS_BROWSER_VERIFICATION=1`) can override the config file setting. When a module is disabled, its registration is skipped, its commands are excluded from the CLI help, and attempts to invoke it produce a clear error message pointing to the configuration option.

## Consequences

- v5 behaves identically to v4 out of the box — backward compatibility is preserved
- Early adopters can test individual modules without enabling the entire suite
- CLI/env overrides allow CI pipelines and test suites to toggle features without modifying config files
- Feature-flagged code paths remain in the codebase, increasing maintenance surface
- Removing a flag later requires cleanup across config schema, CLI parsing, env var handling, and module registration

## Alternatives Considered

- **All on by default:** Breaks backward compatibility with v4 — users upgrading would see new behavior and potentially new errors without opting in.
- **Compile-time flags (preprocessor/conditional imports):** Provide no runtime control — users would need to rebuild to change feature sets, which is impractical for a distributed CLI tool.
