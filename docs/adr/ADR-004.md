# ADR-004: Credential Management

**Status:** Accepted

**Date:** 2026-06-02

## Context

Assessments require credentials for multiple roles (attacker, victim, admin, user). These credentials must be provided securely, support multiple roles distinctly, and be cleared from memory after use to minimize exposure. The solution must be simple enough for v5 while leaving a path to more secure storage.

## Decision

Use a `--creds` CLI flag pointing to a JSON file with role-keyed credentials. The `CredentialStore` class loads the file once at startup, provides typed accessors (`getCredential(role, service)`), and zeroizes all stored values on completion by overwriting strings in memory. The JSON schema enforces a top-level map of role names to credential objects, each containing service-specific entries (e.g., `{"attacker": {"github": {"token": "...", "username": "..."}}}`).

## Consequences

- Simple and explicit — credentials are declared in a single file, not scattered across environment variables
- Zero-clear on completion reduces the window of exposure in process memory
- JSON file on disk persists between runs — users must manage file permissions themselves
- No encryption at rest in v5 — the file is plaintext; users are responsible for access control
- Not suitable for CI/CD pipelines with secret managers — will be addressed in a future ADR

## Alternatives Considered

- **Environment variables:** Not structured enough for multi-role, multi-service credentials. Namespacing (e.g., `ARGUS_ATTACKER_GITHUB_TOKEN`) becomes unwieldy and error-prone.
- **OS keychain integration:** Adds platform-specific dependencies (Keychain, libsecret, wincred) that increase v5 scope and testing complexity. Reserved for a future version.
