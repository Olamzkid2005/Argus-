# ADR-015: Fork Boundary Enforcement

**Status:** Accepted

**Date:** 2026-06-02

## Context

Argus modules may be forked or reused outside the main repository. Without explicit boundaries, modules can develop implicit dependencies on internal symbols from other modules, making extraction difficult and causing architecture erosion over time.

## Decision

Enforce that Argus modules depend only on symbols explicitly exported from `@opencode/runtime`. Any import that references internal module structure via patterns like `../../opencode/*`, `../opencode/*`, or `**/opencode/*` is blocked by a `no-restricted-imports` ESLint rule. Modules that need to share functionality must export it through the `@opencode/runtime` barrel, which serves as the curated public API surface. The rule is checked in CI and fails the build on violation.

## Consequences

- Clear, enforced API surface — `@opencode/runtime` is the single point of entry for cross-module dependencies
- Forking a module requires only the `@opencode/runtime` dependency, not the entire codebase
- ESLint rule provides immediate feedback during development, not just in CI
- Adding a new public symbol requires updating the `@opencode/runtime` barrel — an intentional gating step
- The barrel file may become a bottleneck if many modules need to export symbols — mitigated by keeping exports coarse-grained

## Alternatives Considered

- **Directory-based enforcement:** Groups code into `public/` and `private/` directories by convention but cannot prevent imports of private symbols. Too loose — directories mix public and private concerns.
- **Naming-based enforcement:** Relies on naming conventions (e.g., `_private` prefix) that drift over time as developers forget or bypass conventions. No automated enforcement.
- **No enforcement:** Architecture erodes as modules develop incidental dependencies on internal implementation details of other modules, making future extraction or reuse costly.
