# ADR-008: Future Multi-Agent Architecture

**Status:** Accepted

**Date:** 2026-06-02

## Context

Argus may eventually support multi-agent execution where specialized agents (recon, exploitation, reporting) run in separate processes or on separate machines. This capability is desirable but would significantly increase v5 scope. The architecture must not preclude this future while remaining simple for v5.

## Decision

Reserve multi-agent execution for v6. The v5 interface is `PhaseExecutor`, which abstracts execution behind a common interface. Two implementations exist: `InProcessExecutor` (v5, runs everything in-process) and the future remote executor (v6, delegates to agent processes). The Planner produces a `PhaseExecutionRequest` (a description of what to execute), and the executor returns a `PhaseExecutionResult`. The Planner is completely unaware of which executor implementation is in use — swapping the executor requires no planner changes. This interface boundary is the only preparation made for multi-agent in v5.

## Consequences

- Zero v5 scope increase — multi-agent is a placeholder, not a feature
- Clean interface boundary protects against architecture erosion — Planner cannot accidentally depend on in-process execution details
- No premature optimization — the interface is minimal and may need revision when v6 requirements solidify
- The interface may be wrong — if v6 requirements diverge from the v5 abstraction, refactoring will be needed

## Alternatives Considered

- **Build multi-agent in v5:** Increases scope risk, delays shipping, and requires designing a distributed system before understanding real usage patterns.
- **Build without any future path:** Paints the architecture into a corner — Planner tightly coupled to in-process execution would require significant refactoring to support multi-agent later.
