# ADR-018: Cache Mode Semantics

**Status:** Accepted

**Date:** 2026-06-09

## Context

The Argus scan pipeline caches tool results in Redis to speed up repeated scans against the same targets. However, there are scenarios where caching is undesirable:

- **Fresh scans:** A user wants to re-scan a target and see current results, not cached data from a previous run.
- **Cache warming:** An operator wants to pre-populate the cache with fresh results for a new target that will be scanned multiple times.

The original `WorkerCache` had a single TTL-based model with no ability to skip reads or writes. The same code path (read cache → execute → write cache) ran on every invocation. A user had to wait for TTL expiry or manually flush the cache to get fresh results.

Two distinct cache semantics are needed: "skip everything" (truly fresh scan) and "skip reads, still write" (refresh cached data for the next scan). These must not be conflated.

## Decision

Introduce a `CacheMode` enum with three values, enforced at the **execution layer** (not inside cache.py):

| Mode | Read Cache | Write Cache | Purpose |
|------|-----------|-------------|---------|
| `NORMAL` | ✅ Read | ✅ Write | Standard behavior |
| `NO_CACHE` | ❌ Skip | ❌ Skip | "I want a truly fresh scan, don't touch cache" |
| `REFRESH` | ❌ Skip | ✅ Write | "Refresh cached data with fresh results for the next scan" |

Key architectural rules:

1. **Policy enforcement lives at the execution layer** (`ToolRunner.run()`, `PipelineRouter`), not inside `cache.py`. The cache remains a dumb storage layer with no awareness of execution modes. This keeps `cache.py` testable in isolation and avoids coupling storage logic with execution flow.

2. **The `CacheMode` enum is a first-class parameter** threaded through `ToolRunner.run()`, `run_with_circuit_breaker()`, and all specialized tool methods (`run_nuclei`, `run_naabu`, etc.). It is not a global flag or thread-local — each invocation can have its own mode.

3. **CLI surface** maps directly to enum values:
   - `argus scan <target>` → `NORMAL`
   - `argus scan <target> --no-cache` → `NO_CACHE`
   - `argus scan <target> --refresh-cache` → `REFRESH`

4. **Observability counters** are added to `WorkerCache` to track mode usage:
   - `cache.record_bypass()` — incremented on NO_CACHE
   - `cache.record_refresh()` — incremented on REFRESH
   - These are **counters only** — cache.py does not take any action based on mode

5. **Graceful degradation** — if hint generation during NO_CACHE/REFRESH fails, the scan continues uninterrupted. The mode enforcement is a simple conditional, not a failure point.

## Consequences

- Users can now force fresh scans (`--no-cache`) without waiting for TTL expiry or manually flushing cache
- Operators can pre-warm the cache (`--refresh-cache`) for targets that will be scanned multiple times
- Cache mode usage is observable via metrics, enabling operational insight into how often fresh vs cached scans are requested
- Existing callers are unaffected — `NORMAL` is the default and behaves identically to the pre-Phase-2 behavior
- The execution layer is slightly more complex (every `run()` call site must accept and pass `cache_mode`)
- TypeScript side (`ExecutionOptions.cacheMode`) is defined but not yet wired into CLI flags — will be connected when the CLI flag parser is updated

## Alternatives Considered

- **Global singleton flag:** Would make it impossible to mix cache modes within a single scan (e.g., REFRESH for tool results, NORMAL for advisory queries). Rejected because the plan explicitly requires per-data-type TTL control.

- **Thread-local storage:** Would hide the parameter from the call chain, making it non-obvious which mode applies to a given invocation. Rejected in favor of explicit parameter passing.

- **Cache-level filtering (cache.py checks a global mode):** Would couple storage logic with execution policy, violating separation of concerns. Rejected — cache.py must remain a dumb storage layer.

- **Two separate flags (`--bypass-cache` and `--no-cache-write`):** Adds UI complexity for marginal benefit. The three-mode model (NORMAL/NO_CACHE/REFRESH) covers all known use cases with a single enum.
