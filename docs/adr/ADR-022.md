# ADR-022: Fixture Testing Strategy

**Status:** Accepted (Partially Implemented — Phase 3A/3B complete, Phase 3C/3D deferred)

**Date:** 2026-06-09

## Context

The `argus-workers` test suite had 141 test files at the time of this decision, all heavily mocked (Redis, DB, LLM, subprocess). The `conftest.py` had only 3 simple fixtures (`sample_finding`, `sample_authorized_scope`, `mock_db_connection_string`). There was no CI for Python tests — only `ruff` linting ran on every PR.

This created a gap: changes to the normalization, dedup, or reporting pipeline could be verified only by running the full E2E suite, which required Redis, a database, and real vulnerability targets. Developers relied on manual testing or hoped the mock coverage was sufficient.

A single vulnerability app E2E test was proposed, but the infrastructure to run it (CI with Python test deps) did not exist.

## Decision

Adopt a **phased, incremental approach** to fixture-driven testing:

### Phase 3A: Test Infrastructure (Prerequisite)

Create `requirements-dev.txt` with pinned test dependencies (`pytest`, `pytest-asyncio`, `pytest-timeout`, `pytest-mock`, `pytest-cov`). Add a `python-tests` CI job that installs these deps and runs the existing test suite excluding DB/Redis/E2E-marked tests.

This must be done first because no other testing changes can be validated without CI.

### Phase 3B: Deterministic Static Fixtures

Create `test_fixtures/` with static JSON fixture data representing real tool outputs (SQL injection, XSS, port scan, tech detection, empty scan, mixed severity). These are deterministic — same fixture always produces the same results. Tests exercise the normalization, dedup, reporting, and finding classification pipeline against this data.

No subprocesses, no ports, no network. Tests run in <5 seconds.

### Phase 3C: Minimal Real-Process E2E (Deferred)

One or two tiny Flask apps (~30 lines each) with a single known vulnerability. Tests start the app on a random port, scan it, and verify the expected finding types are detected.

**Phase 3C is deferred.** Phase 3A (test CI) and Phase 3B (static fixtures) were implemented first and have proven value. E2E smoke testing remains planned but has not been scheduled.

### Phase 3D: Future Expansion (Deferred)

Additional vulnerable app fixtures, broader tool coverage (nmap, nuclei, sqlmap fixtures), nightly CI job for full fixture matrix. **Deferred indefinitely** — not scheduled.

## Implementation Status

**Phase 3A (Test Infrastructure):** ✅ Complete — `requirements-dev.txt` created, `lint.yml` has a `python-tests` CI job running existing tests.

**Phase 3B (Static Fixtures):** ✅ Complete — 6 fixture JSONs + 3 error output JSONs in `test_fixtures/`, `conftest.py` helpers (`load_fixture`, `get_fixture_path`), 17 pipeline regression tests in `test_fixture_pipeline.py`.

**Phase 3C (E2E Smoke Tests):** ❌ Deferred — not implemented. Requires real web app fixtures and CI job.

**Phase 3D (Future Expansion):** ❌ Deferred — not scheduled. Will be revisited when Phase 3B maintenance burden is understood.

## Consequences

- Pipeline regression tests run in <5 seconds — fast enough for pre-commit or local development.
- Deterministic fixtures catch regressions that mock-only tests miss (e.g., a field name change in the normalization layer).
- No infrastructure dependencies (no Redis, no DB, no real targets).
- Phase 3C provides the highest value (catches subprocess boundary bugs) but requires more setup — deferred until Phase 3B proves ROI.
- Test CI adds ~2 minutes to PR checks, well within acceptable limits.

## Alternatives Considered

- **Single monolithic fixture suite (skip phases):** Jumping to full E2E before establishing test infrastructure would create flaky CI failures and high maintenance burden. Rejected in favor of incremental value delivery.
- **Mocks-only (status quo):** Misses pipeline-layer bugs that manifest only when real data flows through multiple processing steps. Rejected.
- **Snapshot testing (record/replay real tool output):** Adds complexity for marginal benefit over static JSON fixtures. The tool output format is relatively stable — a snapshot adds little that a well-structured fixture doesn't provide.
