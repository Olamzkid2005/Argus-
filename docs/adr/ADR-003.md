# ADR-003: Planner Decision Mode

**Status:** Accepted

**Date:** 2026-06-02

## Context

The Planner module must classify assessment targets, detect technologies, and select appropriate tools. Relying solely on LLM-based classification provides rich context awareness but fails offline. Relying solely on deterministic rules works offline but misses nuanced, context-dependent findings. A hybrid approach is needed.

## Decision

Use a hybrid decision mode: LLM-first classification via OpenCode providers for target classification and technology detection, with a deterministic regex-based fallback (`planDeterministic.ts`) when the LLM is unavailable. The planner first attempts LLM analysis; if the provider returns an error or times out, it falls back to the deterministic engine. Results from either path conform to the same `PlannedPhase[]` type, so downstream consumers are unaware of which mode produced the plan.

## Consequences

- Best-effort accuracy — LLM provides rich classification when available; deterministic fallback ensures offline operation
- Single code path for execution — both modes produce identically typed output
- Deterministic fallback can miss context-dependent findings that an LLM would catch
- Dual code paths increase maintenance surface — changes to tool capabilities must be reflected in both the LLM prompt context and the regex patterns

## Alternatives Considered

- **Pure LLM:** No offline capability — the tool becomes unusable without network access, violating a core requirement.
- **Pure deterministic:** Misses context-dependent findings (e.g., identifying a custom framework by its unique error-handling patterns), producing lower-quality assessments.
