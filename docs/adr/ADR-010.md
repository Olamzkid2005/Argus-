# ADR-010: Evidence Integrity

**Status:** Accepted

**Date:** 2026-06-02

## Context

Evidence collected during assessments must be verifiable — users need to confirm that artifacts have not been tampered with after collection. The integrity mechanism must be practical for a local CLI tool and not introduce unnecessary complexity.

## Decision

Use SHA256 hashing per artifact as the integrity mechanism for v5. The `EvidenceManifest` struct contains a `package_hash` that covers the entire manifest plus the list of artifact hashes. Each artifact in the package directory has a corresponding `.sha256` sidecar file. The `verifyPackage()` function recomputes all hashes and compares them against the manifest. Mismatches are reported with the specific artifact path and expected-vs-actual hash values. This provides tamper detection without the complexity of cryptographic signatures.

## Consequences

- Simple implementation — SHA256 is available in Python's `hashlib` stdlib, no additional dependencies
- Tamper-evident — any modification to an artifact or the manifest is detected on verification
- Self-contained — all integrity data lives alongside the artifacts in the evidence package
- No authenticity guarantees — SHA256 proves integrity but not origin; a user cannot distinguish between tampering and legitimate replacement without signatures
- Large artifacts require streaming hash computation to avoid loading entire files into memory

## Alternatives Considered

- **Digital signatures (GPG/Ed25519):** Provide authenticity and integrity but require key management, key distribution, and user education. Unnecessary overhead for local-only evidence where the user is both the producer and consumer.
- **Merkle tree:** Provides efficient verification of subsets but is overengineered for the current use case where all artifacts in a package are verified together. Would add implementation complexity without practical benefit for v5.
