# ADR-017: Progress Callback Throttling (Deferred)

**Status:** Deferred

**Date:** 2026-06-04

## Context

The `WorkflowRunner.onProgress` callback streams status updates during assessment execution. Currently, every phase transition and tool execution fires a separate callback with no throttling or batching.

Example call sites in `workflow-runner.ts`:
- Target validation complete
- Engagement created/loaded
- Plan created with N phases
- Each phase start/complete/fail
- Bridge connected/disconnected
- Assessment complete/failed

At present, each workflow runs sequentially with phases that take seconds (WhatWeb, Katana, Nuclei, Browser Verify). The callback frequency is low — typically fewer than 10 updates per assessment.

## Decision

Do not add throttling or batching at this stage. The current callback frequency is well within tolerable limits for the TUI rendering layer, and adding backpressure now would introduce complexity with no measurable benefit.

The `onProgress` callback signature remains `(status: string) => void`. Callers receive every update and must handle frequency themselves if needed.

## Consequences

- **Simpler code** — no debounce timers, batch buffers, or backpressure state to maintain
- **Callers must handle bursts** — if future changes introduce faster workflows or parallel phase execution, TUI consumers may need to add their own throttling
- **No test burden** — the current callback contract is trivial to test and reason about

## When to Revisit

Add throttling or batching when any of these conditions are met:

1. **Parallel phase execution** is introduced — multiple phases could emit overlapping progress bursts
2. **Workflows emit >10 updates/sec** in production — measured telemetry, not hypothetical
3. **TUI rendering becomes a bottleneck** — profiling shows progress updates competing with frame rendering

## Alternatives Considered

- **Add debounce now:** Would require a configurable debounce window, batch accumulator, and flush-on-complete logic. Premature optimization — adds code that may never be exercised.
- **Batch updates per-phase:** Phase-level batching (emit once per phase instead of per-step) loses granularity needed for the live scan dashboard.
